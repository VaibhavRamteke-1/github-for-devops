#Git Real Project Workflow (End-to-End) :
Understand how Git is used in a real DevOps project lifecycle from development to production.

#Branch Structure :
main      → Production (live code)
develop   → Staging / integration
feature/* → New features
hotfix/*  → Urgent production fixes

#Step-by-Step Workflow :
1) Developer creates feature branch.
git checkout develop
git pull origin develop
git checkout -b feature/login

Work is always done on feature branch (not on master)

2) Developer writes code & commits.
git add .
git commit -m "Add login functionality" 

3) Push feature branch.
git push origin feature/login

4) Create Pull Request (PR).
From: feature/login
To: develop
Code review happens here

5) Merge after approval.
Usually via Git platform (PR merge)
Result:
develop branch updated with feature

6) CI/CD pipeline triggers
After merge:
Build runs
Tests run
Code deployed to staging

7) Testing on QA environment
QA team validates feature
Bugs identified

8) Fixes (if needed)
git checkout feature/login
# fix code
git commit -m "Fix login bug"
git push
PR updates automatically, no need to create new PR request.
QA team validates again

9) Release to production
git checkout main
git merge develop
git push origin main
Production deployment triggered

##Hotfix Workflow (Production Issue) :

Step 1: Create hotfix branch
git checkout main
git checkout -b hotfix/payment-fix

Step 2: Fix issue
git commit -m "Fix payment bug"
git push origin hotfix/payment-fix

Step 3: Merge to master
git checkout master
git merge hotfix/payment-fix
git push

Step 4: Merge back to develop
git checkout develop
git merge hotfix/payment-fix
git push
Keeps branches in sync

#Key Points :
Never commit directly to master
Always use feature branches
Use PR for code review 
CI/CD automates build & deploy
